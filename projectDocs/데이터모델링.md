# 🗄️ Database Schema & Security Design (Phase 1)

## 1. 테이블 설계 (Tables)

### 1.1. `profiles` (사용자 프로필)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | uuid | PK, FK (auth.users) | 사용자 고유 ID |
| `username` | text | Unique, Not Null | 사용자 닉네임 |
| `avatar_url` | text | | 프로필 이미지 경로 |
| `is_discoverable` | boolean | Default: true | 연락처 기반 추천 허용 여부 |
| `updated_at` | timestamp | | 수정 일시 |

### 1.2. `follows` (팔로우 관계)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `follower_id` | uuid | FK (profiles.id) | 팔로우를 신청한 사람 |
| `following_id` | uuid | FK (profiles.id) | 팔로우 대상자 |
| `status` | text | 'pending', 'accepted' | 팔로우 수락 상태 |
| `created_at` | timestamp | | 팔로우 시점 |
| *Composite PK* | (follower_id, following_id) | | 중복 팔로우 방지 |

### 1.3. `custom_groups` (사용자 정의 그룹)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | uuid | PK | 그룹 고유 ID |
| `owner_id` | uuid | FK (profiles.id) | 그룹 생성자 |
| `name` | text | Not Null | 그룹 이름 (예: 친한 친구) |

### 1.4. `group_members` (그룹 멤버 매핑)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `group_id` | uuid | FK (custom_groups.id) | 대상 그룹 |
| `user_id` | uuid | FK (profiles.id) | 그룹에 포함된 팔로워 |

### 1.5. `records` (자서전 기록)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | uuid | PK | 기록 고유 ID |
| `user_id` | uuid | FK (profiles.id) | 작성자 ID |
| `question` | text | Not Null | 질문 내용 |
| `answer` | text | Not Null | 답변 내용 |
| `visibility` | enum | 'private', 'mutual', 'group', 'public' | 공개 범위 |
| `created_at` | timestamp | | 작성 일시 |

### 1.6. `record_group_access` (기록-그룹 공유 매핑)
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `record_id` | uuid | FK (records.id) ON DELETE CASCADE | 대상 기록 |
| `group_id` | uuid | FK (custom_groups.id) ON DELETE CASCADE | 공유할 그룹 |
| *Composite PK* | (record_id, group_id) | | 중복 공유 방지 |

## 2. 보안 정책 (Supabase RLS Policy)

### 2.1. `records` 테이블 조회(SELECT) 권한
작성자가 아닌 사용자가 기록을 보려면 다음 조건 중 하나를 만족해야 함:
1. `visibility = 'public'`
2. `visibility = 'mutual'` 이고, 서로 `follows` 테이블에 `status = 'accepted'` 상태로 존재함 (맞팔로우).
3. `visibility = 'group'` 이고, `auth.uid()`가 해당 기록과 연결된 `record_group_access` 내의 `group_id`에 속해 있음 (`group_members` 테이블 참조).

### 2.2. 연락처 해시 매칭 (Security)
- 사용자의 연락처 정보를 서버에 그대로 저장하지 않고, `sha256(phone_number)` 값을 별도 테이블에 임시 저장하여 가입자 간 해시값 대조를 통해서만 친구 추천을 수행함.

## 3. SQL 예시 (RLS 초안)
```sql
-- 맞팔로우(Mutual) 권한 체크를 위한 Policy 예시
CREATE POLICY "Mutual followers can view shared records" 
ON records FOR SELECT 
USING (
  visibility = 'mutual' AND 
  EXISTS (
    SELECT 1 FROM follows f1
    JOIN follows f2 ON f1.follower_id = f2.following_id AND f1.following_id = f2.follower_id
    WHERE f1.follower_id = auth.uid() AND f1.following_id = records.user_id
    AND f1.status = 'accepted' AND f2.status = 'accepted'
  )
);
